# HQ 0.3 — Business Ops Template

> **Clone a real business OS. Define your domain in code. Ship the rest.**

**Release date:** 2026-04-18
**Tag:** `v0.3.0`
**License:** MIT

---

HQ 0.3 is the release where HQ becomes genuinely useful as a starting point.
Fork it, register your objects, and you inherit the foundation every
agentic internal system needs: identity, permissions, objects, actions,
events, workflows, agents, jobs, inbox, files, integrations,
import/export, MCP, diagnostics, and a Workshop UI — all behind one
registry pattern, one policy engine, and one Zod-validated action
dispatcher.

Sixteen planned workstreams landed in this release. Two removable
example modules (CRM, Projects/Tasks) prove the architecture end to end
without turning HQ into a project-management product.

## At a glance

- **Registry-driven Object Studio** — register a Prisma model once, get
  list/detail/form/API/MCP/permissions for free.
- **Unified permission model** — one `can()` engine covers users, bots,
  agents, and MCP clients uniformly.
- **Central action dispatcher** — every write (Workshop, API, workflows,
  agents, MCP) runs through the same Zod-validated, scope-gated,
  approval-aware pipeline.
- **Activity timeline** — every mutation, action, workflow step, agent
  turn, approval, and file event lands in `platform_events` and streams
  via `pg_notify`.
- **Agent governance + MCP parity** — actions expose scopes, approvals,
  and schemas uniformly. MCP tools are generated from registered
  actions.
- **Two removable example modules** — CRM (Customer, Product) and
  Projects/Tasks with canonical-User ownership.
- **Builder DX** — four `add-*.md` quickstarts plus runnable templates
  for objects, actions, workflows, agents, and full modules.
- **Ops hardening** — jobs, inbox, integrations, import/export, files
  with pluggable storage, diagnostics, `pnpm doctor`, production deploy
  scripts.

## What's new

### Object Studio — the core registry

Register a Prisma model once, and Workshop + API + MCP + permissions
all light up for free.

```ts
// shared/objects/src/modules/billing.ts
export const billingObjects: Record<string, ObjectDefinition> = {
  Invoice: {
    type: 'Invoice',
    label: 'Invoice',
    labelPlural: 'Invoices',
    icon: 'receipt',
    schema: invoiceSchema,         // Zod
    fields: {
      number:   { kind: 'text',   label: 'Number',    required: true },
      customer: { kind: 'ref',    label: 'Customer',  ref: 'Customer' },
      amount:   { kind: 'number', label: 'Amount',    currency: 'USD' },
      status:   { kind: 'enum',   label: 'Status',    options: INVOICE_STATUSES },
      dueDate:  { kind: 'date',   label: 'Due' },
      contract: { kind: 'file',   label: 'Contract' },
    },
    sort:    { field: 'dueDate', order: 'asc' },
    filters: ['status', 'customer'],
    actions: ['markPaid'],         // custom action, registered separately
  },
};
```

Out of the box you get:

- `/objects/Invoice` list + filter + search + detail + create + edit
- Auto-registered CRUD actions (`Invoice.list`, `.get`, `.create`,
  `.update`, `.delete`, `.bulk_delete`, `.bulk_update`)
- API routes at `/v1/objects/Invoice/*`
- MCP tool schemas for every action
- Policy checks at every boundary (`can(principal, 'objects.update', invoice)`)
- Activity timeline entries for every write

Docs: [objects.md](../objects.md) · [add-object.md](../add-object.md)

### Unified permission model

One engine for every principal type:

```ts
import { can } from '@hq/auth';

if (await can(principal, 'invoices.refund', invoice)) { … }
```

- Users, service bots, AI agents, MCP clients — same call, same vocabulary.
- `PermissionKey` spans settings/identity/agents/workflows/approvals/actions/audit.
- Object ownership resolves per-resource.
- `GET /v1/me/permissions` returns the resolved capability set + per-object access map.

Docs: [permissions.md](../permissions.md)

### Central action dispatcher

Every action — CRUD-generated or custom — runs through the same path:

```ts
defineAction({
  name: 'invoice.markPaid',
  title: 'Mark invoice paid',
  objects: ['Invoice'],
  scopes: ['invoices.update'],
  approval: { required: (input) => input.amount > 10_000 },
  parameters: z.object({ invoiceId: z.string(), paidAt: z.string().datetime() }),
  handler: async (ctx, { invoiceId, paidAt }) => {
    const invoice = await markPaid(ctx, invoiceId, paidAt);
    await emitEvent(ctx, 'invoice.paid', { objectType: 'Invoice', objectId: invoiceId });
    return invoice;
  },
});
```

Call it from anywhere and you get the same guarantees:

| Caller | Gets |
| --- | --- |
| Workshop UI | Zod validation, policy check, approval gate, audit event |
| `POST /v1/actions/invoice.markPaid` | Same |
| Workflow `action` node | Same |
| Agent tool call | Same |
| MCP client tool invocation | Same — auto-derived tool schema |

Docs: [actions.md](../actions.md) · [add-action.md](../add-action.md)

### Agent governance + MCP parity

Agents call the same dispatcher as Workshop, workflows, and MCP. High-risk
actions hit the shared approval queue; scope grants apply uniformly
whether the caller is human, bot, or MCP.

```ts
defineAgent({
  name: 'collections-bot',
  skills: ['invoicing'],
  capabilities: ['invoices.read', 'invoices.sendReminder'],
  // no 'invoices.refund' — requires approval for this agent
});
```

The MCP server generates its `tools/list` response from registered
actions, with real JSON schemas derived from each action's Zod
parameters. External agents get the same action surface as internal
callers.

Docs: [agents.md](../agents.md) · [mcp.md](../mcp.md) · [add-agent.md](../add-agent.md)

### Deterministic workflows

Workflows are code, rendered as a graph. Nine node types: `action`,
`agent`, `function`, `condition`, `delay`, `loop`, `parallel`,
`subworkflow`, `wait`. Four trigger types: `manual`, `cron`, `event`,
`webhook`.

```ts
defineWorkflow({
  key: 'invoices.collect-overdue',
  triggers: [{ type: 'cron', cron: '0 9 * * 1' }],
  entryNodeId: 'find',
  nodes: [
    { id: 'find', type: 'action', actionName: 'Invoice.list',
      inputMap: { status: 'overdue' } },
    { id: 'remind', type: 'action', actionName: 'invoice.sendReminder',
      forEach: '{{steps.find.output.data}}',
      inputMap: { invoiceId: '{{item.id}}' } },
  ],
  evals: {
    remind: async (_, output) => [{
      name: 'reminders_sent',
      passed: (output as any).remindersSent >= 0,
      score: 1,
    }],
  },
});
```

Docs: [workflows.md](../workflows.md) · [add-workflow.md](../add-workflow.md)

### Activity timeline (events)

Every mutation, action, workflow step, agent turn, approval, and file
event lands in `platform_events`. An in-process `subscribe()` bus plus
`pg_notify` covers both enrichment hooks and streaming UIs.

```ts
import { subscribe } from '@hq/events/router';

subscribe('file.created', async (event) => {
  const ctx = await makeSystemContext();
  await enqueueThumbnail(ctx, event.objectId);
}, { source: 'thumbnailer' });
```

Open catalog — no central enum. Emit a new event from a handler, it
just works.

Docs: [events.md](../events.md)

### Filesystem-shaped files

Folders + files + nested paths. Pluggable `StorageAdapter`:

```bash
STORAGE_DRIVER=local        # dev, streams through API
STORAGE_DRIVER=s3           # prod — S3/R2/MinIO, presigned uploads
```

Three field types for referencing files from any object:

```ts
{ kind: 'file',   label: 'Contract' }
{ kind: 'files',  label: 'Attachments', min: 0, max: 8 }
{ kind: 'folder', label: 'Working directory' }
```

`TEMP` folders with `retentionDays` get swept automatically by the
`files.sweep-temp` job.

Docs: [files.md](../files.md)

## Upgrading from 0.2

0.2 shipped an internal skeleton. 0.3 changes enough that the intended
upgrade path is: **fork 0.3 fresh, migrate your business objects into
the new registry, then delete the CRM + Projects modules**.

If you're on a 0.2 checkout you care about:

| 0.2 | 0.3 |
| --- | --- |
| Bespoke list/detail pages per model | Register an `ObjectDefinition`; Workshop auto-generates |
| Hand-rolled `checkPermission` calls | `can(principal, 'resource.action', resource?)` |
| Routes invoking services directly | Route → dispatcher → registered action |
| `password` required on `User` | Nullable; `IdentityAccount` for SSO |
| No event log | Every mutation emits to `platform_events` (side effect, don't emit manually) |
| No approval queue | Actions declare `approval.required`; queue surfaces in Workshop |
| Hardcoded file uploads | `file`/`files`/`folder` field types + pluggable storage |

Before upgrading a production database: see
[operations/backup-restore.md](../operations/backup-restore.md). The
migration chain from 0.2 is non-trivial; treat it as a new install and
import your data.

## Known limitations

These are intentional choices, not regressions. Each has a decision
doc behind it.

- **No visual workflow builder.** Workflows are code. Workshop renders
  the graph; editing lives in your editor.
- **Single-tenant.** Run one deployment per tenant. Multi-tenant was
  deferred to avoid a half-built abstraction.
- **No SAML or SCIM.** OIDC covers the common cases; add a SAML provider
  if you need one — the abstraction exists.
- **No integration marketplace.** The framework is there; each module
  owns its own integrations. A discovery surface is 0.4+ territory.
- **No hosted cloud.** Self-hosted only. No billing layer.
- **Screenshots in docs** reference Workshop surfaces but are captured
  at release-cut time — see [SMOKE-CHECKLIST.md](./SMOKE-CHECKLIST.md)
  for the shot list.

## What's next — 0.4

Tentatively scoped around **hardening existing foundations** and
**raising the agent ceiling**, not adding new registries.

- Operational dashboard (object counts, workflow health, agent status,
  next-step hints).
- Workflow evals as a first-class surface: `/evals` treats pass/fail as
  a product metric.
- Agent skills marketplace convention — community-shared skills loaded
  into an HQ install safely.
- Richer MCP: object resources + action tools + prompt catalog.
- Streaming logs + replay in diagnostics.
- Canonical cron catalog — every scheduled job visible in Workshop with
  a last-run/next-run timeline.

Follow the [roadmap issue](https://github.com/aiwah-labs/hq/issues) for
the live version.
