# HQ 0.3 — Business Ops Template

**Released:** 2026-04-18

## Headline

Clone HQ, define your business in code, and get the foundation every agentic
internal system needs: identity, permissions, objects, actions, events,
workflows, agents, jobs, inbox, files, integrations, import/export, MCP,
diagnostics, and Workshop UI — all behind one registry pattern, one policy
engine, and one Zod-validated action dispatcher.

0.3 turns HQ from a credible skeleton into a cloneable business OS template.
Sixteen plans landed: five registries unified behind a single dispatcher,
two removable example modules (CRM, Projects/Tasks), a functioning
Workshop, and the builder docs + templates to add your own vertical in an
afternoon.

## What changed

### Foundations

- **[Object Studio](../objects.md)** — register a Prisma model once, get
  list/detail/form/API/MCP/permissions for free. Auto-generated
  `<type>.list/.get/.create/.update/.delete/.bulk_*` actions. File/files/folder
  field types. Extended metadata (labels, icons, sort, filters, empty states).
- **[Identity & SSO](../identity.md)** — canonical `User` + `IdentityAccount`
  with a provider abstraction. Generic [OIDC starter](../sso.md)
  (Google/Okta/Azure/Authentik/Keycloak/Auth0), env-driven domain/role
  mapping, dependency-free implementation.
- **[Unified Permission Model](../permissions.md)** — one `can()` engine covers
  users, bots, agents, MCP clients. `PermissionKey` vocabulary spans
  settings/identity/agents/workflows/approvals/actions/audit. Policy checks
  wired into object CRUD, the action dispatcher, and Workshop routes.
  `/v1/me/permissions` returns resolved capabilities + per-object access map.
- **[Template Module Boundaries](../modules.md)** — explicit split between
  `platformObjects` and `moduleObjects`; seeds under `seed-modules/`; each
  example module ships an adapt-and-remove checklist.

### Example modules (removable)

- **[CRM](../example-modules/crm.md)** — Customer and Product, migrated to
  the registry; lives in `shared/objects/src/modules/crm.ts`.
- **[Projects & Tasks](../example-modules/projects.md)** — `Project` +
  `Task` with canonical-`User` ownership and assignment, portfolio view,
  blocked-tasks widget, activity timeline. Proves the architecture end to
  end; safe to rip out.

### Agents, workflows, actions

- **[Agent Governance + MCP Parity](../agents.md)** — agents call the same
  dispatcher as Workshop/API/workflows. Actions expose scopes, approvals,
  and schemas uniformly. [`docs/mcp.md`](../mcp.md) covers the MCP surface
  generated from registered actions.
- **[Activity Timeline](../events.md)** — every mutation, action,
  workflow step, agent turn, approval, and file event lands in
  `platform_events`. `pg_notify` streams to Workshop + diagnostics;
  in-process `subscribe()` handles enrichment.

### Operations

- **[Diagnostics](../../apps/workshop/src/app/(app)/diagnostics/)** — health
  endpoint, env warnings, recent failures, event replay.
- **[Setup & Deploy](../../README.md)** — rewritten README + DEPLOY.md,
  `pnpm doctor`, expanded `.env.example`, `scripts/setup-server.sh` +
  `scripts/first-deploy.sh` + `scripts/setup-nginx.sh`.
- **[Jobs & Scheduling](../../shared/jobs/src/)** — `enqueueJob`,
  `cancelJob`, `listJobRuns`, Workshop jobs page. `pg-boss` under the hood.
- **[Inbox & Notifications](../../apps/workshop/src/app/(app)/inbox/)** —
  `InboxItem` model + service + API + Workshop UI; approvals auto-surface
  in the queue.
- **[Integrations Framework](../integrations.md)** — provider-agnostic
  shape: static credentials, OAuth, per-connection ACL, secret storage.
- **[Import / Export](../import-export.md)** — CSV/JSON export for every
  registered object; preview-validate-commit import flow.
- **[Files & Folders](../files.md)** — filesystem-shaped model (folders
  + files, nested paths, USER/SYSTEM/TEMP kinds), pluggable
  `StorageAdapter` (local + S3/R2/MinIO), presigned + passthrough upload,
  `files.sweep-temp` lifecycle job, `file`/`files`/`folder` field types.

### Builder DX

- **[Builder quickstarts](../building-with-hq.md)** — five-registry map,
  four `add-*.md` guides (object/action/workflow/agent), conventions.
- **[Runnable templates](../../templates/)** — single-file skeletons for
  each registry, plus a full `templates/module/` bundle (objects + seed +
  actions) for anyone scaffolding a new vertical.

## Upgrade notes (from 0.2)

0.2 shipped an internal-only skeleton. 0.3 changes enough that the only
sensible upgrade path is **fork 0.3, migrate your business objects into
the new registry, then delete the CRM + Projects modules**.

If you are on a 0.2 checkout you cared about:

- **Objects** moved to the registry pattern. Your Prisma models still
  work, but Workshop CRUD is now driven by `registerObject(...)` entries,
  not bespoke pages. See [`docs/add-object.md`](../add-object.md).
- **Auth** is nullable-password + `IdentityAccount`. Existing users keep
  working; SSO is opt-in via `AUTH_OIDC_*` env vars.
- **Permissions** changed shape. If you hand-rolled policy checks in
  service code, replace them with `can(principal, permission, resource?)`
  from [`shared/auth/src/policy.ts`](../../shared/auth/src/policy.ts).
- **Actions** now go through the central dispatcher. A route or agent
  call that previously invoked a service function directly should call
  the registered action by name — that's how you inherit policy,
  approvals, audit, and MCP exposure.
- **Events** are now a required side effect of every mutation. Don't
  patch service functions to emit events manually; rely on
  `shared/objects/src/crud.ts` and service-layer helpers.

Production operators: see [`docs/operations/backup-restore.md`](../operations/backup-restore.md)
for the Postgres dump + uploaded-files + secrets checklist before
upgrading anything you care about.

## Known limitations

- **Visual workflow builder** — intentionally out of scope. Workflows are
  code. Workshop renders the graph, but editing lives in your editor.
- **Multi-tenant orgs** — single-tenant by design. Run one deployment per
  tenant, isolate at the infrastructure layer.
- **SCIM / full SAML** — not shipped. OIDC covers the common cases; a
  SAML adapter is a drop-in provider if you need one.
- **Integration marketplace** — the framework is there; there is no
  discovery surface or shared registry. Each module owns its own
  integrations.
- **Billing / hosted cloud** — not included. HQ is self-hosted only.
- **Screenshots in docs** — the doc index references Workshop surfaces;
  screenshots are captured at release-cut time (see
  [`SMOKE-CHECKLIST.md`](./SMOKE-CHECKLIST.md)).

## Next release direction

0.4 is scoped around **making existing foundations production-hard** and
**raising agent ceiling**, not adding new registries.

Candidates:

- Operational dashboard (deferred from Plan 1 Step 8) — object counts,
  recent records, workflow health, agent status, next-step hints.
- Workflow evals as a first-class surface: a `/evals` page that treats
  eval pass/fail as a product metric, not an afterthought.
- Agent skills marketplace convention — how a community-shared skill
  gets loaded into an HQ install safely.
- Richer MCP surface: object resources + action tools + prompt catalog.
- Streaming logs + replay in diagnostics.
- Canonical cron catalog — every scheduled job visible in Workshop with
  a last-run/next-run timeline.
